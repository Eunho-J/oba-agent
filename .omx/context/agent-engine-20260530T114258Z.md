# Context Snapshot: Agent Engine

## Task Statement

Plan the first private-repo implementation for a lightweight OBA agent engine that supports MCP and tool calls.

## Desired Outcome

Define a concrete implementation plan, architecture boundary, and verification path for a minimal agent engine that can:

- Use a short personality-style system prompt.
- Expose only four built-in coding tools initially: `read`, `write`, `edit`, `bash`.
- Treat tools as evolvable extension artifacts without implementing tool self-evolution yet.
- Use `codex-as-api` with existing Codex OAuth credentials as a temporary testbed provider.
- Preserve a future path to OpenRouter and EXAONE/FriendlyAI.
- Provide structured logging and full traceback/error output.
- Avoid arbitrary fallbacks, mocks, and shallow smoke-only tests.

## Known Facts / Evidence

- Current repo is private dev repo at `/Users/cayde/Workspace/oppa/oba-agent`.
- Existing server is `services/agent-gateway`, currently a thin HTTP router around providers and adapters.
- Existing tests are `node --test services/agent-gateway/test/*.test.js`.
- Official Codex Rust implementation exposes a tool-router pattern: model-visible specs, a registry, tool-call construction, and dispatch. Relevant upstream files inspected:
  - `openai/codex` `codex-rs/core/src/tools/router.rs`
  - `openai/codex` `codex-rs/core/src/tools/registry.rs`
  - `openai/codex` `codex-rs/core/src/tools/handlers/mod.rs`
- Official Codex docs say Codex supports MCP servers in CLI/IDE, with configuration under `mcp_servers.<id>` including `command`, `url`, `enabled_tools`, approval modes, and timeouts.
- Official Codex auth docs say CLI supports ChatGPT sign-in and API-key sign-in, and caches credentials under `~/.codex/auth.json` or OS keyring.
- `codex-as-api` exists on npm as `codex-as-api@0.4.0`, a third-party OpenAI-compatible local API server backed by ChatGPT/Codex OAuth. It exports `ChatGPTOAuthProvider` and `createApp` and exposes `/v1/chat/completions`.
- Pi Coding Agent docs identify the default four tools as `read`, `write`, `edit`, and `bash`; additional read-only tools exist but are not default.

## Constraints

- Planning only under ralplan. No implementation edits until consensus handoff is complete and the user selects an execution lane.
- Keep the engine very light; do not add framework bulk.
- Do not include `codex-as-api` in release/runtime as a committed production dependency unless explicitly approved later.
- No mock provider as a substitute for real provider behavior.
- No arbitrary fallback paths. Provider failure should be explicit.
- Logs must support debugging, including traceback/stack output.
- Tool evolution is an architectural extension point only in the first implementation.

## Unknowns / Open Questions

- Exact shape of future FriendlyAI/EXAONE API compatibility.
- Resolved in PRD: MCP v1 uses `@modelcontextprotocol/sdk@1.29.0` as the only new runtime dependency for MCP client support.
- Resolved in PRD: v1 uses `codex-as-api` as an external local OpenAI-compatible HTTP server, not by importing `ChatGPTOAuthProvider`.
- Resolved in PRD: tool filesystem and bash access are restricted to a single configured workspace root, defaulting to the repository root.
